Key Risks for Ralph in Large Codebases

Coder Autonomy

An autonomous agent picking tasks and implementing without deep architectural context may create patterns conflicting with existing ones.

Reachability Model

Designed for new consumer-facing features, not refactors or bug fixes. In brownfield, you're often modifying existing features.

git commit --amend

In shared repos with team members, amending commits on rework is risky. Force-push conflicts with collaborative workflows.

Token Cost at Scale

4-6x multiplier across multiple developers adds up fast at enterprise scale.

Large codebase verdict: Superpowers is safer. Its advisory model lets developers maintain control while getting discipline benefits. Ralph's autonomous execution is powerful but risky when there are established conventions, shared branches, and architectural constraints requiring human judgment.

6 Deployment Recommendation

Use both, layered. Superpowers as the always-on foundation; Ralph for scoped feature work on isolated branches.

Layered Strategy

LayerToolScopeConstraints
Foundation Superpowers All projects, all developers Low-risk, low-cost. Enforces TDD, design-first, evidence-before-claims without autonomous action.
Feature Work Ralph Well-scoped new features on feature branches Always HITL mode in shared codebases. Pre-populate copilot-instructions.md with real conventions. Feature branches only, never main.
Gap Fill Superpowers skills During Ralph execution Developers invoke debugging/TDD skills when Ralph's Coder hits issues. Fills Ralph's debugging and discipline gaps.

7 Internal Patterns Without MCP

Six approaches for supplying internal library knowledge and coding conventions to AI assistants, ranked by effort and effectiveness.

1. copilot-instructions.md

Zero effort Always-on

Auto-loaded by Copilot on every request. Put the top 5-10 critical rules here.

## Internal Libraries

### @company/auth
- Import: `import { authenticate, requireRole } from '@company/auth'`
- NEVER roll custom auth. Always use this library.
- `authenticate(req)` returns `AuthContext` or throws `UnauthorizedError`
- `requireRole('admin')` is Express middleware

## Patterns

### API Endpoints
- All endpoints in `src/api/{resource}/router.ts`
- Use `createRouter()` from `@company/api-framework`
- Validation via zod schemas in `src/api/{resource}/schema.ts`
Pros: Zero setup, always available, works with any AI tool.
Cons: ~4KB effective before it dilutes attention. Static.

2. Scoped Instructions

VS Code Copilot Contextual

In .github/instructions/ — loaded only when editing matching files.

# .github/instructions/api.instructions.md
---
applyTo: "src/api/**/*.ts"
---
# API Conventions
- Use `createRouter()` not `express.Router()`
- Every route needs `authenticate` middleware
- Response shape: `{ data, meta, errors }`
- Always validate with zod schema before processing
# .github/instructions/testing.instructions.md
---
applyTo: "**/*.test.ts"
---
# Testing Conventions
- Use `createTestContext()` from `@company/test-utils`
- Database tests use `withTestDb()` wrapper for cleanup
- Never mock `@company/auth` — use `createTestUser()`
Pros: Contextual — only loaded for relevant files. Scales better than a single file.
Cons: VS Code Copilot specific.

3. Example Files as Living Documentation

Moderate effort Testable

Canonical examples the AI reads as real code. Reference from instructions.

docs/examples/

Pros: AI reads real code, not descriptions. Examples are testable and stay accurate.
Cons: Must maintain. AI must be told to look at them.

4. Per-Directory Convention Files

Moderate effort Discoverable

Drop CONVENTIONS.md in directories with strong local patterns.

# src/api/CONVENTIONS.md
- Each resource gets its own directory
- router.ts exports a single `createXxxRouter()` function
- schema.ts contains all zod schemas for this resource
- Never import from sibling resources — use events
Pros: Discoverable by both AI and humans. Close to the code it governs.
Cons: Not auto-loaded by Copilot (Claude Code loads CLAUDE.md). Can go stale.

5. Library Type Stubs

TypeScript LSP-powered

For internal libraries the AI can't install, create declaration files the LSP picks up.

// src/types/company-libs.d.ts
declare module '@company/auth' {
  export function authenticate(req: Request): Promise<AuthContext>;
  export function requireRole(role: string): Middleware;
  export interface AuthContext {
    userId: string;
    roles: string[];
    org: string;
  }
}
Pros: TypeScript LSP picks these up. AI gets full type info without the package.
Cons: Must keep in sync with real library. TypeScript only.
